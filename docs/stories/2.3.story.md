# Story 2.3: Consulta de pedidos por repartidor

## Status

Draft

## Story

**As a** auxiliar de mostrador,
**I want** consultar qué números de pedido tiene registrados un repartidor específico,
**so that** pueda resolver rápido la duda de quién se llevó un pedido determinado.

## Acceptance Criteria

1. Desde el Resumen de Domicilios, se puede seleccionar un repartidor y ver el listado de números de pedido registrados a su nombre.
2. El listado incluye la fecha/hora de registro de cada pedido.
3. La consulta no requiere buscar en WhatsApp ni en ningún registro externo al sistema.

  - [ ] `app/api/repartidores/[id]/pedidos/route.ts` (o lectura directa desde Server Component, ver Nota): retorna `RegistroSalida[]` del repartidor con `numeroPedido` y `registradoEn` [Source: architecture/api-specification.md#rest-api-specification]
- [ ] Task 2: Componente DetalleRepartidor (AC: 1, 2)

  - [ ] Test de integración: dado un repartidor con 2 registros, la consulta retorna ambos con su `numeroPedido` y `registradoEn` [Source: architecture/testing-strategy.md#test-organization]

## Dev Notes

### Previous Story Insights

[Source: docs/stories/2.2.story.md] La pantalla de Resumen ya lista repartidores con cantidad y valor a pagar; esta historia añade el nivel de detalle (drill-down) por repartidor, reutilizando el mismo patrón de Server Component sin estado de cliente.

### Component Specification

[Source: architecture/components.md#detallerepartidor-server-component]

- **Responsibility:** Lista los números de pedido registrados bajo un repartidor específico.
- **Key Interfaces:** Ruta `/resumen/[repartidorId]`.
- **Dependencies:** `lib/db/registros.ts`.
- **Technology Stack:** React Server Component.

### Routing

[Source: architecture/frontend-architecture.md#routing-architecture]

```text
/resumen/[id]        - Detalle de pedidos por repartidor
```

### API Specification

[Source: architecture/api-specification.md#rest-api-specification]

`GET /repartidores/{id}/pedidos` → 200 con `RegistroSalida[]` (incluye `numeroPedido` y `registradoEn` de cada uno). Dado que `DetalleRepartidor` es un Server Component, la implementación puede optar por consultar `lib/db/registros.ts` directamente en el servidor en lugar de pasar por este endpoint HTTP — ambas opciones son válidas según el patrón BFF implícito de la arquitectura [Source: architecture/high-level-architecture.md#architectural-patterns]; se mantiene el endpoint documentado por si se necesita consumo externo en el futuro.

### Data Models

[Source: architecture/data-models.md#registrosalida]

Cada `RegistroSalida` expone `numeroPedido` y `registradoEn` — ambos campos requeridos por el AC 2 de esta historia.

### File Locations

- `app/resumen/[repartidorId]/page.tsx` (nuevo)
- `app/api/repartidores/[id]/pedidos/route.ts` (nuevo, opcional según decisión de implementación arriba)

[Source: architecture/unified-project-structure.md]

### Testing

[Source: architecture/testing-strategy.md#test-organization]

Test de integración verificando que el detalle por repartidor retorna exactamente los registros de ese repartidor (y no los de otros), con número de pedido y fecha/hora.
