# Story 2.3: Consulta de pedidos por repartidor

## Status

Draft

## Story

**As a** auxiliar de mostrador,
**I want** consultar qué números de pedido tiene registrados un repartidor específico,
**so that** pueda resolver rápido la duda de quién se llevó un pedido determinado.

## Acceptance Criteria

1. Desde el Resumen de Domicilios, se puede seleccionar un repartidor y ver el listado de números de pedido registrados a su nombre.
2. El listado incluye la fecha/hora de registro de cada pedido.
3. La consulta no requiere buscar en WhatsApp ni en ningún registro externo al sistema.

> **Nota de implementación (actualizada tras validación de mockups):** el diseño original de esta historia asumía una página separada (`/resumen/[repartidorId]`). El mockup validado con el propietario (`mockups/mockup2.png`) muestra este detalle como un **acordeón expandible dentro de la misma pantalla `/resumen`** — el AC arriba sigue siendo válido tal cual ("se puede seleccionar un repartidor y ver el listado"), pero la implementación cambia: no hay navegación ni cambio de URL. Ver `docs/front-end-spec.md`.

## Dev Notes

### Previous Story Insights

[Source: docs/stories/2.2.story.md] La pantalla de Resumen ya lista repartidores con cantidad y valor a pagar, y cada fila ya está preparada para renderizar un `RepartidorAccordionRow`. Esta historia implementa el contenido de ese acordeón — no agrega una ruta ni una navegación nueva.

### Component Specification

[Source: architecture/components.md#repartidoraccordionrow-client-component]

- **Responsibility:** Fila expandible de un repartidor dentro de `ResumenDomicilios`; al expandir, muestra los números de pedido y fecha/hora registrados a su nombre.
- **Key Interfaces:** Recibe los pedidos del repartidor como prop desde `ResumenDomicilios` (mismo fetch del Server Component padre, sin round-trip HTTP adicional).
- **Dependencies:** `components/ui/accordion` (shadcn).
- **Technology Stack:** React Client Component (necesita estado local de expandido/colapsado).

### Routing

[Source: architecture/frontend-architecture.md#routing-architecture]

```text
/resumen            - Resumen de Domicilios (incluye detalle por repartidor como acordeón inline, sin ruta propia)
```

No existe `/resumen/[id]` — fue removida de la arquitectura tras la validación de mockups.

### Mockup validado (fuente de verdad visual)

[Source: docs/front-end-spec.md#resumen-de-domicilios] Ver `mockups/mockup2.png`: Carlos aparece expandido mostrando "Pedidos de Carlos" con número + hora por fila; Diego y Andrés aparecen colapsados. Pendiente confirmar con el propietario si el acordeón es exclusivo (uno a la vez) o si varios pueden expandirse simultáneamente — implementar como independientes (no exclusivo) por defecto si no hay respuesta antes de esta historia, por ser el comportamiento más simple de construir.

### API Specification (opcional)

[Source: architecture/api-specification.md#rest-api-specification]

`GET /repartidores/{id}/pedidos` → 200 con `RegistroSalida[]` (incluye `numeroPedido` y `registradoEn`). Este endpoint queda documentado pero es OPCIONAL para esta historia: al ser el acordeón parte del mismo Server Component que ya cargó todos los datos del resumen, no hay necesidad de un round-trip HTTP adicional para expandir una fila [Source: architecture/high-level-architecture.md#architectural-patterns].

### Data Models

[Source: architecture/data-models.md#registrosalida]

Cada `RegistroSalida` expone `numeroPedido` y `registradoEn` — ambos campos requeridos por el AC 2 de esta historia.

### File Locations

- `app/api/repartidores/[id]/pedidos/route.ts` (nuevo, opcional)

[Source: architecture/unified-project-structure.md]
