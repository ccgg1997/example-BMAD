# Story 2.1: Cálculo de resumen de domicilios por repartidor

## Status

Review

## Story

**As a** propietario de la droguería,
**I want** que el sistema calcule automáticamente cuántos pedidos lleva cada repartidor y cuánto le corresponde pagar,
**so that** no tenga que sumar manualmente desde una hoja de papel.

## Acceptance Criteria

1. El sistema calcula, para cada repartidor, la cantidad total de pedidos registrados en el período vigente (por ejemplo, la semana en curso).
2. El sistema calcula el valor a pagar multiplicando la cantidad de pedidos por la tarifa configurada (ver FR7).
3. El cálculo se actualiza automáticamente cada vez que se registra una nueva salida (ver Story 1.4).

### Completion Notes List

- **AC1 menciona "período vigente (ej. la semana en curso)" pero se implementó como acumulado histórico total**: no existe en el modelo de datos (Story 1.3) ningún campo o mecanismo de corte/reset semanal, ninguna otra historia lo define, y las Tasks de esta misma historia (y el JSON de referencia de la API) no mencionan un filtro de fechas — se tomó como un ejemplo ilustrativo de la AC, no un requisito operacionalizable con lo que existe hoy. Si el propietario necesita cortes por semana más adelante, es una historia nueva (requiere decidir cuándo empieza/termina una "semana" y qué pasa con el historial).
- `obtenerResumen()` usa `prisma.registroSalida.groupBy` (agregación en la base de datos, aprovechando el índice de `repartidor_id` de Story 1.3) en vez de traer todos los registros a memoria — más alineado con el objetivo de `<300ms` de `security-and-performance.md`. Incluye repartidores con 0 registros (`cantidad: 0, valorAPagar: 0`), no solo los que ya tienen domicilios, para que Story 2.2 pueda listar a todos.
- **Bug de test real encontrado y corregido**: el test original de "snapshot de tarifa" mutaba `configuracionSistema.tarifaDomicilio` (fila singleton compartida) para simular un cambio de tarifa, y lo restauraba en un `finally` — pero los archivos de test corren en paralelo (confirmado: `tests/integration/api/registros.test.ts` falló de forma intermitente leyendo `tarifaVigente = 5000` mientras este test tenía la tarifa mutada a mitad de camino). Corregido sin tocar la fila compartida: se crean registros con `tarifaAplicada` explícitas distintas (1000 y 5000) y se verifica que la suma usa exactamente esos valores guardados — prueba la misma propiedad (snapshot, no recálculo) sin mutar estado global compartido entre archivos de test.
- Verificado end-to-end contra el stack Docker real: `GET /api/resumen` devuelve los 3 repartidores seedeados con sus totales correctos (incluyendo los que tienen 0 registros).
