# Story 2.1: Cálculo de resumen de domicilios por repartidor

## Status

Draft

## Story

**As a** propietario de la droguería,
**I want** que el sistema calcule automáticamente cuántos pedidos lleva cada repartidor y cuánto le corresponde pagar,
**so that** no tenga que sumar manualmente desde una hoja de papel.

## Acceptance Criteria

1. El sistema calcula, para cada repartidor, la cantidad total de pedidos registrados en el período vigente (por ejemplo, la semana en curso).
2. El sistema calcula el valor a pagar multiplicando la cantidad de pedidos por la tarifa configurada (ver FR7).
3. El cálculo se actualiza automáticamente cada vez que se registra una nueva salida (ver Story 1.4).

## Dev Notes

### Previous Story Insights

[Source: docs/stories/1.4.story.md, docs/stories/1.5.story.md] `RegistroSalida` ya se crea con `tarifaAplicada` como snapshot desde el flujo de registro; esta historia solo agrega la capa de agregación, no modifica la escritura.

### Component Specification

[Source: architecture/components.md#resumenservice-libservicesresumence]

- **Responsibility:** Agrega registros por repartidor y calcula el valor total a pagar.
- **Key Interfaces:** `obtenerResumen(): ResumenPorRepartidor[]`
- **Dependencies:** `lib/db/registros.ts`

### Data Models

[Source: architecture/data-models.md#registrosalida]

`RegistroSalida.tarifaAplicada` es el campo a sumar (no recalcular con la tarifa vigente) — así lo exige la regla de snapshot [Source: architecture/coding-standards.md#critical-fullstack-rules].

### API Specification

[Source: architecture/api-specification.md#rest-api-specification]

`GET /api/resumen` → 200:
```json
[
  { "repartidor": { "id": "...", "nombre": "Carlos", "activo": true }, "cantidad": 12, "valorAPagar": 12000 }
]
```

### Performance

[Source: architecture/security-and-performance.md#performance-optimization]

- Response Time Target: <300ms bajo el volumen esperado (decenas de registros/día).
- Database Optimization: usar los índices existentes en `repartidor_id` y `registrado_en` para la agregación [Source: architecture/database-schema.md].
- Sin capa de caché — el volumen de uso no la requiere en el MVP.
