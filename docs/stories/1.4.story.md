# Story 1.4: Pantalla de Registro de Salida

## Status

Review

## Story

**As a** repartidor,
**I want** seleccionar mi nombre e ingresar los números de los pedidos que me llevo,
**so that** quede registrado rápidamente qué pedidos tengo, sin llenar formularios largos.

## Acceptance Criteria

1. La pantalla permite seleccionar el nombre del repartidor de una lista, sin escribir texto libre.
2. La pantalla permite agregar uno o varios números de pedido antes de confirmar.
3. Al presionar "Registrar salida", los pedidos ingresados quedan guardados con el repartidor y la fecha/hora actual.
4. El flujo completo (seleccionar repartidor + ingresar pedidos + confirmar) toma menos de 15 segundos en un caso típico de 1 a 3 pedidos (ver NFR1).
5. La interfaz es usable desde un navegador móvil (ver NFR2).
6. La pantalla solo es accesible con sesión válida (ver Story 1.2); sin sesión, redirige a login.

### Data Access Layer (referencia)

[Source: architecture/backend-architecture.md#data-access-layer]

```typescript
// lib/db/registros.ts
export async function crearRegistros(repartidorId: string, numeros: number[]) {
  const tarifa = await obtenerTarifaVigente();
  const creados = [];
  const conflictos = [];
  for (const numeroPedido of numeros) {
    try {
      creados.push(
        await prisma.registroSalida.create({
          data: { repartidorId, numeroPedido, tarifaAplicada: tarifa },
        })
      );
    } catch (e) {
      const existente = await prisma.registroSalida.findUnique({ where: { numeroPedido } });
      conflictos.push({ numeroPedido, asignadoA: existente?.repartidorId });
    }
  }
  return { creados, conflictos };
}
```

_Nota:_ esta misma función implementa también la validación de doble asignación de Story 1.5 — el manejo de `conflictos` en el catch ya está cubierto aquí; Story 1.5 se enfoca en la UX de mostrar ese conflicto al usuario.

### Conectividad Intermitente (NFR3)

[Source: architecture/checklist-results-report.md#risk-assessment-top-5] El PRD (NFR3) exige tolerar conexión intermitente propia de un punto de venta de barrio. El diseño original de `apiPost` no incluía reintentos; esta historia es la responsable de agregarlos (ver Task 6) porque es el único punto donde el repartidor envía datos mutables. La idempotencia queda garantizada gratis por la restricción `UNIQUE` de `numero_pedido` (Story 1.3): reintentar un envío ya guardado no duplica, solo reporta un "conflicto" contra sí mismo.

### Completion Notes List

- **`app/registro/page.tsx` necesitó `export const dynamic = "force-dynamic"`**: sin esto, Next.js prerenderiza la página en build time (detectado en el output del build: `○ /registro` en vez de `ƒ`), horneando la lista de repartidores como HTML estático — un repartidor nuevo o desactivado después del deploy no se reflejaría hasta el próximo build. Confirmado el fix viendo el output cambiar a `ƒ (Dynamic)`.
- `lib/db/registros.ts#crearRegistros` implementado calcando el snippet de referencia de `architecture/backend-architecture.md` tal cual: cualquier error de `prisma.registroSalida.create` (no solo la violación de la restricción UNIQUE) se reporta como "conflicto". Probado manualmente con un `repartidorId` inexistente y confirmado que también cae en la rama de conflicto (con `asignadoA: null`) en vez de propagar un error real — **no se corrigió aquí a propósito**: Story 1.5 tiene como Task 2 explícito "Manejo de conflicto por número en `crearRegistros`", que es precisamente donde corresponde discriminar el código de error de Prisma (`P2002` = violación UNIQUE real) de cualquier otro fallo.
- Se omitió `lib/services/registro.ts` como archivo separado (la historia lo mencionaba junto a `lib/db/registros.ts` con "/", ambigüedad de nombre): `app/api/registros/route.ts` llama `crearRegistros` directo desde `lib/db/registros.ts`, sin una capa de servicio intermedia que no agregaría lógica propia — evita un archivo vacío de puro paso.
- `lib/tarifa.ts` nuevo (no mencionado explícitamente en "File Locations" de esta historia, pero sí en `unified-project-structure.md`): aísla `obtenerTarifaVigente()` para que Story 2.1/2.2 lo reutilicen sin duplicar la consulta a `configuracion_sistema`.
- `lib/api-client.ts#apiPost` (Task 6, NFR3): reintenta con backoff (300ms, 600ms, ...) solo ante fallo de red o 5xx; un 4xx se propaga de inmediato sin reintentar (es un error permanente de la solicitud). La idempotencia ante reintento queda cubierta gratis por la restricción UNIQUE de `numero_pedido` (Story 1.3): un reintento de un envío ya guardado aparece como "conflicto" contra sí mismo, no lo duplica.
- Verificado end-to-end contra el stack Docker real: login → `GET /api/repartidores` devuelve los 3 seedeados → `GET /registro` con sesión responde 200 (y sin sesión, 307 a `/login`, confirmando AC6) → `POST /api/registros` crea registros reales con `tarifaAplicada` = tarifa vigente.
