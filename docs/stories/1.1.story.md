# Story 1.1: Configuración base del proyecto y despliegue

## Status

InProgress

## Story

**As a** propietario de la droguería,
**I want** que exista una aplicación web desplegada y accesible desde un celular,
**so that** el equipo pueda empezar a usar el sistema desde el primer día sin depender de instalaciones.

## Acceptance Criteria

1. El repositorio del proyecto está creado con la estructura de proyecto único Next.js definida en Technical Assumptions y Architecture.
2. La aplicación está desplegada vía Docker Compose (app + PostgreSQL + Caddy como reverse proxy con HTTPS) en un entorno accesible por URL desde un navegador móvil.
3. Existe un endpoint de verificación (`/api/health`) que confirma que la aplicación y su conexión a base de datos están funcionando.
4. La base de datos relacional está aprovisionada, conectada a la aplicación, y con las migraciones iniciales de Prisma aplicadas.
5. El pipeline de CI/CD (GitHub Actions) está configurado: corre pruebas en cada push y despliega automáticamente a producción al hacer merge a main.
6. Los frameworks de testing (Vitest + React Testing Library) están instalados y configurados, con al menos una prueba de ejemplo pasando.
7. Tailwind CSS y shadcn/ui están instalados y configurados como base de estilos para las siguientes historias.

## Dev Notes

### Previous Story Insights

No hay historia previa — esta es la primera historia del proyecto.

### Tech Stack

[Source: architecture/tech-stack.md#technology-stack-table]

- Frontend + Backend unificados: Next.js (App Router) 15.x, TypeScript 5.x
- Database: PostgreSQL 16.x
- CSS: Tailwind CSS 4.x + shadcn/ui (sobre Radix)
- Testing: Vitest + React Testing Library
- Build/Bundler: Next.js CLI (Turbopack dev / Webpack prod)
- IaC: Docker Compose v2
- CI/CD: GitHub Actions

### Project Structure

[Source: architecture/unified-project-structure.md]

Proyecto único Next.js (NO monorepo multi-paquete): `app/` (rutas UI + `api/`), `components/` (incluye `ui/` de shadcn), `lib/` (db, services, auth, config), `prisma/`, `tests/` (`unit/`, `integration/`), `middleware.ts`, `docker-compose.yml`, `Dockerfile`, `Caddyfile`, `.env.example`.

### Deployment

[Source: architecture/deployment-architecture.md#deployment-strategy]

- Frontend y backend se despliegan juntos, mismo contenedor Docker (Next.js standalone output) — no hay CDN/edge separado.
- Método de deploy: GitHub Actions → SSH al VPS → `docker compose pull && docker compose up -d`.
- Pipeline de referencia [Source: architecture/deployment-architecture.md#cicd-pipeline]: job `test` (`npm ci`, `npm run test`) seguido de job `deploy` (needs: test) que hace SSH y `docker compose up -d --build`.
- Ambiente único de Producción; sin Staging separado para este MVP [Source: architecture/deployment-architecture.md#environments].

### Environment Variables

[Source: architecture/development-workflow.md#environment-configuration]

```bash
DATABASE_URL=postgresql://user:pass@localhost:5432/domicilios
APP_PASSWORD=clave-compartida-del-negocio
SESSION_SECRET=una-cadena-aleatoria-larga
SENTRY_DSN=                       # opcional
```

### Coding Standards relevantes

[Source: architecture/coding-standards.md#critical-fullstack-rules]

- Variables de entorno centralizadas: acceder solo vía `lib/config.ts`, nunca `process.env` disperso por el código.

### Riesgos de arquitectura a tener en cuenta en esta historia

[Source: architecture/checklist-results-report.md#risk-assessment-top-5]

- Postgres no debe exponer su puerto fuera de la red interna de Docker Compose; firewall del Droplet debe permitir solo 80/443/22.
- Fijar versiones exactas de shadcn/ui y Vitest en `package.json` al instalar (evitar `latest` sin congelar).
- Sin backup automatizado de PostgreSQL, un fallo del Droplet borra todo el historial de domicilios y pagos — por eso el Task 8 de esta historia no es opcional.

### Testing

[Source: architecture/testing-strategy.md]

- Frameworks: Vitest + React Testing Library.
- Ubicación: `tests/unit/` y `tests/integration/`.
- Esta historia solo requiere una prueba de ejemplo pasando para validar que el framework está bien configurado; la cobertura funcional llega en historias siguientes.
